# TaskManagePlus 온보딩 플로우 (최적화 버전)

TaskManagePlus는 **태스크 자동 생성 및 관리**와 **메모리뱅크(지식/회고/진행 로그) 시스템**이 통합된 프로젝트 관리 솔루션입니다.

온보딩 플로우는 아래와 같이 두 시스템이 자연스럽게 연동되도록 최적화되어 있습니다.

---

## 온보딩 시 필수 안내: 개발 단계별 품질 점검 및 일관성 관리

> **TaskManagePlus를 사용하는 모든 프로젝트는 아래의 품질 점검 및 일관성 관리 플로우를 반드시 숙지하고 실천해야 합니다.**
>
> - **개발 시작:** 린트/포맷팅, 네이밍, 테스트/문서화 등 품질 기준을 확인하고 체크리스트를 참고해 설계/구현을 시작하세요.
> - **중간(복잡도 높은 코드 구현 직후):** Cyclomatic Complexity 10 이상, 100줄 이상 등 복잡한 코드 작성 후 즉시 점검(린트/정적분석/테스트/리뷰) 및 기준 미달 시 즉시 보완/리팩토링을 권장합니다.
> - **개발 종료(릴리즈/완료):** 전체 코드/모듈/태스크 완료 시, 린트/테스트/정적분석/복잡도/문서화 등 모든 품질 기준을 점검하고, 미달 항목은 즉시 수정/보완/리팩토링 후 완료 처리하세요.
> - 각 단계별 체크리스트/템플릿을 반드시 활용하고, memory-bank/05-progress-log.md 등에도 점검 결과와 개선점을 기록하세요.

---

## ✅ 품질 관리 기준(클린코드 및 자동화 기준)

TaskManagePlus는 프로젝트의 품질을 높이기 위해 아래와 같은 기준을 적극적으로 권장합니다.

### 1. 클린코드 원칙

- 명확하고 일관된 네이밍
- 함수/클래스의 단일 책임 원칙
- 중복 최소화
- 불필요한 주석/코드 제거
- 적절한 예외 처리 및 오류 관리
- 테스트 가능한 구조
- 코드의 의도를 코드 자체로 표현
- (자세한 내용은 `docs/clean-code-java-src/` 참고)

### 2. 객관적/자동화 가능한 품질 관리 기준

- **코드 린트/포맷팅**: ESLint, Prettier, Black 등 도구로 자동 검사 및 포맷팅
- **테스트 커버리지**: 단위/통합 테스트 작성, 커버리지 측정(예: 80% 이상)
- **코드 리뷰/PR 규칙**: 모든 주요 변경은 PR로 제출, 1인 이상 리뷰 필수
- **정적 분석 도구**: SonarQube, CodeClimate 등으로 코드 복잡도/보안/버그 탐지
- **의존성/취약점 관리**: npm audit, Snyk 등으로 취약점 자동 점검
- **문서화 수준**: API 문서 자동화(Swagger 등), 주요 구조/다이어그램 최신화
- **CI/CD 자동화**: 빌드, 테스트, 배포 자동화(GitHub Actions 등)
- **이슈/태스크 관리**: 모든 작업은 태스크/이슈로 관리, 상태/우선순위/담당자 명확화
- **회고/피드백 기록**: 태스크/프로젝트 단위로 회고 및 개선점 기록
- **버전/릴리즈 관리**: Git Flow, SemVer 등 일관된 브랜치/버전 관리

### 3. 실제 적용 예시(체크리스트)

- 태스크 템플릿(예: `templates/task-template.md`)에 아래와 같은 품질 체크리스트를 추가해 활용하세요.

```
- [ ] 클린코드 원칙 준수
- [ ] 린트/포맷팅 통과
- [ ] 테스트 작성 및 통과
- [ ] 코드 리뷰 완료
- [ ] 문서화(README/API 등) 최신화
- [ ] 주요 변경/회고 memory-bank 연동
```

- memory-bank/05-progress-log.md에 품질 관련 주요 이슈, 개선점, 회고를 타임라인 형태로 기록

## 코드 품질 점검 및 일관성 관리 플로우

TaskManagePlus는 개발 전 과정에서 코드 품질과 일관성을 체계적으로 관리할 수 있도록 아래와 같은 프로세스를 권장합니다.

---

## 1. 개발 시작 시점 안내

- 프로젝트/모듈/태스크 시작 시, 린트/포맷팅, 네이밍, 테스트/문서화 등 품질 기준을 반드시 확인하세요.
- 품질 체크리스트/템플릿을 참고하여 설계 및 초기 구현을 시작하세요.

## 2. 중간 점검(복잡도 높은 코드 구현 직후)

- Cyclomatic Complexity 10 이상, 100줄 이상 등 복잡도가 높은 함수/클래스/모듈 구현 직후 즉시 점검을 권장합니다.
- 린트/정적분석/테스트/리뷰 등 품질 점검을 실행하고, 기준 미달(복잡도 초과, 린트 오류, 테스트 미통과 등) 시 즉시 코드 분리, 네이밍 개선, 중복 제거, 리팩토링 등 보완을 권장합니다.

## 3. 개발 종료(릴리즈/태스크 완료) 시점 안내

- 전체 코드/모듈/태스크 완료 시, 아래 항목을 반드시 점검하세요.
  - 린트/포맷팅 오류 0개
  - 테스트 커버리지 기준 충족(예: 80% 이상)
  - 정적 분석 주요 이슈 0개
  - 코드 복잡도/중복/문서화 등 선택 지표 기준 충족
  - 품질 체크리스트/회고/피드백 기록 완료
- 기준 미달 항목이 있으면 즉시 수정/보완/리팩토링 후 완료 처리하세요.

## 4. 점검 결과에 따른 자동 제안 예시

- 복잡도/품질 기준 미달 시: "⚠️ 코드 복잡도가 기준(10) 초과(12)입니다. 함수 분리, 네이밍 개선, 중복 제거 등 리팩토링을 권장합니다."
- 테스트 커버리지 미달 시: "⚠️ 테스트 커버리지가 기준 미달입니다. 추가 테스트 작성 후 재점검하세요."
- 린트 오류 발견 시: "⚠️ 린트 오류 발견. 코드 스타일/네이밍 규칙을 준수해 수정하세요."
- 기준 충족 시: "✅ 모든 품질 기준을 충족했습니다. 다음 단계(리뷰/릴리즈/회고)로 진행하세요."

## 5. 실제 적용 예시(템플릿/자동화 메시지)

### 개발 시작

```
[개발 시작 체크리스트]
- [ ] 린트/포맷팅 규칙 확인
- [ ] 네이밍/파일 구조 규칙 확인
- [ ] 테스트/문서화 기준 확인
```

### 복잡도 높은 코드 구현 직후

```
[중간 점검]
- [ ] Cyclomatic Complexity: 12 (기준 10 초과) → 함수 분리/리팩토링 권장
- [ ] 린트 오류: 0개
- [ ] 테스트 통과: O
```

### 개발 종료/릴리즈 전

```
[종료 점검]
- [ ] 린트 오류: 0개
- [ ] 테스트 커버리지: 85%
- [ ] 정적 분석 주요 이슈: 0개
- [ ] 코드 복잡도: 평균 8.2
- [ ] 품질 체크리스트/회고 기록 완료
```

- 기준 미달 시: "⚠️ [종료 점검] 기준 미달 항목 발견 → 즉시 수정/보완/리팩토링 후 완료 처리하세요."

## 1단계: 기획서/요구사항 입력

- Notion, 파일, 링크, 외부 시스템 등 다양한 방식으로 프로젝트 기획서/요구사항을 입력합니다.
- (MCP 연동, 파일 첨부, URL 입력 등)

### ➡️ [신규] 1.5단계: 기획서 내용 자동 분류 및 메모리 뱅크 템플릿 반영

- 입력한 기획서/요구사항이 존재하면, memory-bank 템플릿(00-project-overview.md, 01-architecture.md 등)에 주요 내용을 **자동으로 분류/입력**합니다.
- 자동화 도구/스크립트/AI가 기획서의 각 항목(목표, 구조, 아키텍처, API 등)을 해당 템플릿 파일에 맞게 채워줍니다.
- 주요 항목 예시:
  - 프로젝트 목표/범위/구조 → 00-project-overview.md
  - 시스템 아키텍처/설계 패턴/데이터 흐름 → 01-architecture.md
  - 주요 모듈/외부 API/확장 구조 → 02-components.md
  - 개발/배포/리뷰/테스트 프로세스 → 03-development-process.md
  - API 명세/인증/에러 코드 → 04-api-documentation.md
  - 초기 진행 로그/결정/회고 → 05-progress-log.md
- 자동화가 불가한 경우에만 온보딩 안내에 따라 직접 입력할 수 있습니다.
- 이 단계에서 메모리뱅크의 각 문서가 실제 프로젝트 맥락에 맞게 채워지므로, 이후 회고/분석/지식 관리가 훨씬 쉬워집니다.

## 2단계: 태스크 자동 생성 및 파일명 규칙 선택

- 입력된 기획서를 바탕으로 주요 태스크가 자동 분할/추천됩니다.
- 온보딩 과정에서 태스크 파일명 규칙(주제명만/우선순위+주제명)을 선택합니다.
- 선택한 규칙에 따라 `tasks/` 폴더에 태스크가 개별 파일로 생성됩니다.
- 태스크 항목(필드)은 템플릿을 기준으로 하되, 언제든 자유롭게 커스터마이즈할 수 있습니다.

## 3단계: 태스크 진행/상태/회고 관리

- 생성된 태스크의 상태, 우선순위, 담당자, 진행 로그, 회고/피드백 등을 관리합니다.
- 소요시간에 영향을 준 주요 원인 등도 기록해 태스크 최적화에 활용합니다.

## 4단계: 메모리뱅크(Memory Bank) 설정 및 연동

- 태스크 관리와 동시에, 프로젝트의 주요 결정, 구조, 진행 상황, 회고, 학습 내용 등을 `memory-bank/` 폴더에 체계적으로 기록합니다.
- 회고/피드백, 진행 로그 등은 태스크와 메모리뱅크 양쪽에 연동/참조될 수 있습니다.
- 대규모 프로젝트는 도메인별로 하위 디렉토리 분리도 가능합니다.

### [메모리뱅크 온보딩 안내]

- memory-bank 폴더에는 아래와 같은 파일이 기본 제공됩니다:
  - `00-project-overview.md`: 프로젝트 목표/범위/주요 구조/시작 가이드
  - `01-architecture.md`: 시스템 아키텍처, 설계 패턴, 데이터 흐름, 기술 결정 등
  - `02-components.md`: 주요 모듈/서비스/외부 API/확장 구조 등
  - `03-development-process.md`: 작업 흐름, 브랜치 전략, 코드 리뷰, 배포 등
  - `04-api-documentation.md`: 엔드포인트, 요청/응답 예시, 인증, 에러 코드 등
  - `05-progress-log.md`: 날짜별/버전별 주요 변경 사항, 구현 연대기, 회고 등
- 각 파일은 실제 프로젝트 상황에 맞게 자유롭게 확장/수정하세요.
- 예시: `05-progress-log.md`에는 태스크 회고/피드백, 주요 결정, 학습 내용 등을 타임라인 형태로 기록

#### [사용자가 온보딩에서 직접 지정/설정할 수 있는 항목]

- memory-bank의 구조(도메인별 하위 폴더, 추가 파일 등)
- 각 문서의 작성/업데이트 책임자
- 기록/갱신 주기(예: 주요 이벤트 발생 시, 주간/월간 등)
- 작성 방식 및 상세 수준(예시, 코드, 다이어그램 등)
- 진행 로그/회고 기록 방식(자동/수동, 요약/원문 등)
- 공유/접근 권한(협업/외부 공개 등)
- 템플릿/포맷 커스터마이즈(항목 추가/삭제/수정)
- 위 항목들은 온보딩 단계에서 자유롭게 설정할 수 있으며, 운영 중에도 언제든 변경 가능합니다.

### [태스크와 메모리뱅크 연동 구조 및 예시]

- TaskManagePlus에서는 태스크(Task)와 메모리뱅크(Memory Bank)가 유기적으로 연동되어 프로젝트의 실행과 기록, 회고가 자연스럽게 이어집니다.

#### 연동 흐름 예시

1. **태스크 생성/진행**
   - 요구사항/기획서를 바탕으로 태스크가 자동 생성되고, 진행/상태/회고/피드백이 기록됩니다.
2. **태스크 회고/피드백 작성**
   - 각 태스크 완료 시, 회고/피드백(실제 소요 시간, 난이도, 느낀 점, 소요시간 영향 요인 등)을 기록합니다.
3. **메모리뱅크로 연동**
   - 태스크의 회고/피드백, 주요 결정, 학습 내용 등을 `memory-bank/05-progress-log.md`에 요약/참조/복사하여 타임라인 형태로 기록합니다.
   - 필요시 태스크 파일에서 memory-bank의 관련 문서(아키텍처, 컴포넌트, API 등)로 직접 링크/참조할 수 있습니다.
4. **통합 회고/분석**
   - memory-bank의 진행 로그, 구조/결정/회고 기록과 태스크 데이터를 함께 분석하여 프로젝트의 개선점, 주요 인사이트, 통계 리포트 등을 도출할 수 있습니다.

#### 실제 예시

- 태스크 파일의 회고/피드백 섹션에 작성한 내용을 아래와 같이 memory-bank/05-progress-log.md에 연동:


## 태스크 파일 네이밍 규칙(온보딩에서 선택)

- 온보딩 시 아래 두 가지 파일명 규칙 중 하나를 선택할 수 있습니다.
- 언제든 운영 중에도 자유롭게 변경 가능합니다.

### 1. 기본 파일명 규칙

- `tasks/task-주제명.md`
- 예시: `tasks/task-로그인.md`, `tasks/task-회원가입.md`, `tasks/task-커뮤니티-CRUD.md`

### 2. 우선순위 포함 파일명 규칙

- `tasks/task-우선순위-주제명.md` (우선순위는 숫자)
- 예시: `tasks/task-1-로그인.md`, `tasks/task-2-회원가입.md`, `tasks/task-3-커뮤니티-CRUD.md`

## 기존 태스크 파일 일괄 변경 자동화

- 이미 생성된 태스크 파일이 있다면, 아래 스크립트로 파일명을 일괄 변경할 수 있습니다.
- 예시(Bash):

```bash
# 우선순위가 태스크 파일 내에 "- 우선순위: N" 형식으로 있을 때
cd tasks
for file in task-*.md; do
  prio=$(grep "- 우선순위:" "$file" | awk '{print $3}')
  base=$(echo "$file" | sed 's/^task-//;s/\.md$//')
  mv "$file" "task-$prio-$base.md"
done
```

- 반대로, 우선순위 없는 파일명으로 되돌릴 때는 파일명에서 숫자만 제거하면 됩니다.

> 자동화 스크립트는 프로젝트 상황에 맞게 커스터마이즈해 사용하세요.

## 태스크 생성/관리 규칙

- 모든 태스크는 `tasks/` 폴더에 **개별 파일**(`task-*.md`)로 생성/관리합니다.
- 태스크 파일명은 위에서 선택한 규칙에 따라 작성합니다.
- 템플릿은 `templates/` 폴더에 위치하며, 신규 태스크 생성 시 복사해 사용합니다.
- 태스크 상태/진행/회고/피드백 등은 각 태스크 파일 내에서 관리합니다.
- **모든 날짜/시간 기록은 시스템의 현재 시간 또는 온라인에서 서울(Asia/Seoul) 기준 시각을 사용합니다.**

## 태스크 생성 방법

1. `templates/` 폴더에서 `task-template.md` 파일을 복사합니다.
2. `tasks/` 폴더에 선택한 파일명 규칙에 따라 파일명을 지정해 붙여넣습니다.
3. 태스크 제목, 설명, 우선순위, 상태, 담당자, 생성일 등 주요 항목을 작성합니다.
   - **날짜/시간은 시스템 시간 또는 온라인 서울 기준 시각을 참고해 기록합니다.**
4. 진행 로그(History), 회고/피드백(Feedback) 등은 태스크 진행에 따라 파일 내에 추가 기록합니다.

## 예시

```
tasks/
├── task-로그인.md
├── task-회원가입.md
├── task-커뮤니티-CRUD.md

또는

├── task-1-로그인.md
├── task-2-회원가입.md
├── task-3-커뮤니티-CRUD.md
```

## 참고

- 태스크 파일은 Git으로 버전 관리되며, 변경 이력/사유를 자동으로 추적할 수 있습니다.
- 태스크별로 세부 진행상태, 회고, 실제 소요 시간, 난이도 등을 기록해 프로젝트 회고/분석에 활용하세요.
- 템플릿은 자유롭게 커스터마이즈 가능합니다.
- **날짜/시간 기록 시 시스템 시간 또는 온라인 서울 기준 시각을 반드시 확인하세요.**

## 태스크 기록 항목 커스터마이징 안내

- 태스크에 기록해야 할 내용(항목, 형식 등)은 사용자가 자유롭게 커스터마이즈할 수 있습니다.
- 기본 제공 항목과 구조는 `templates/task-template.md` 파일을 기준으로 합니다.
- 필요에 따라 항목을 추가/삭제/수정하여 프로젝트에 맞게 활용하세요.
